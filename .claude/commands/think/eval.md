執行結構化操作評估流程。評估「這個操作要不要自動化、要怎麼實作」，產出統一格式的評估紀錄到 `contexts/thought_review/`。

## 隱式觸發

當對話中出現以下訊號時，SHOULD 主動建議「這個問題適合用 /think:eval 評估，要跑嗎？」，而非直接給臨時建議：

- 「要不要寫成腳本」「可以自動化嗎」「適合自動化嗎」
- 「建 skill 嗎」「要做成 command 嗎」「要不要加進系統」
- 「這樣做有副作用嗎」「會影響什麼嗎」
- 「這個操作值得嗎」「這樣划算嗎」

隱式觸發**不強制**——使用者可以拒絕並繼續對話。

---

## 評估流程

### Step 1 — 確認評估主題

若使用者未明確給出主題，先詢問：「要評估的操作是什麼？」

確認後，在對話中宣告：「開始評估：<主題>」

### Step 2 — 副作用調查

調查此操作的影響範疇，至少涵蓋：

- **直接副作用**：執行此操作會改變什麼？（檔案、狀態、外部服務）
- **失敗副作用**：操作失敗或中斷時，會留下什麼中間狀態？
- **依賴副作用**：是否有其他操作依賴此操作的輸出或狀態？
- **可逆性**：操作是否可輕易撤銷？

### Step 3 — 腳本 vs Agent 判斷

根據以下矩陣判斷實作形式偏好：

| 特徵 | 偏向腳本 | 偏向 Agent 指令 |
|------|---------|----------------|
| 輸入結構 | 固定、可預測 | 模糊、需詮釋 |
| 執行環境 | cron / CI / 無人值守 | 對話中按需執行 |
| 錯誤處理 | 需精確控制 | 可接受 agent 判斷 |
| 重複頻率 | 高（每日/每週） | 低（偶發） |
| 上下文依賴 | 低 | 高 |

### Step 4 — 實作形式決策

從以下選項選擇最適合的實作形式（可複選）：

- **腳本**（`infra/tools/` 或 `infra/periodic_jobs/`）：有固定輸入輸出、需無人值守執行
- **Agent 指令**（`rules/` 或 `.claude/commands/`）：需要對話上下文或 agent 判斷
- **系統 Skill**（`rules/skills/`）：跨 agent 可複用、需要封裝工作流知識
- **文件/範例**（`contexts/` 或 `openspec/`）：僅需記錄最佳實踐，不需自動化
- **混合**：說明各部分的分工

### Step 5 — 命名空間評估（若提案涉及新 command）

若決策包含新增 slash command，評估歸屬：

- `ctx:`：架構開發工具（ctx:arch、ctx:project 等）
- `opsx:`：OpenSpec 設計流程工具
- `think:`：日常使用框架時的思考輔助工具
- 無命名空間：跨 repo 通用工具，適合放 `~/.claude/commands/`

### Step 6 — 輸出評估紀錄

將評估結果寫入 `contexts/thought_review/YYYY-MM-DD_<topic-kebab>.md`，使用以下模板：

```markdown
---
type: eval
date: YYYY-MM-DD
topic: <主題一行摘要>
decision: <script|agent-instruction|skill|docs|hybrid|none>
---

## 問題背景

<描述觸發這次評估的情境與操作>

## 副作用調查

<直接副作用、失敗副作用、依賴副作用、可逆性>

## 判斷框架

<腳本 vs Agent 判斷矩陣的應用結論>

## 決策結論

<選擇的實作形式及理由；若涉及新 command，包含命名空間歸屬決策>

## 實作建議

<下一步具體行動，或「不需實作，記錄即可」>
```

---

## 注意事項

- `think:eval` **只做評估，不執行任何實作**
- 若決策涉及新增 skill，建議後續使用 `create-skill` 工作流
- 若決策涉及架構變更，建議後續使用 `/ctx:arch` 工作流
- 評估紀錄由 observer 透過 `contexts/MANIFEST.md` 的 `thought_review` extraction_focus 識別 `type: eval` 條目
